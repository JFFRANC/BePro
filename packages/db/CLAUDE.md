# BePro DB — Drizzle ORM + Neon PostgreSQL

## Stack
- **ORM:** Drizzle ORM 0.44
- **Database:** Neon PostgreSQL (serverless)
- **Driver:** `drizzle-orm/neon-http` (HTTP, stateless)
- **Migrations:** `drizzle-kit` (generate + push)

## Schema Conventions
- Tables: snake_case plural (`users`, `refresh_tokens`)
- Columns: snake_case (`first_name`, `created_at`)
- All tables include: `id` (uuid PK), `created_at`, `updated_at`, `is_active`
- Tenant-scoped tables include: `tenant_id` (uuid FK → tenants)
- Unique constraints scoped to tenant: `(tenant_id, email)` on users

## Schema Location
Each table is defined in `src/schema/{table-name}.ts` and exported via `src/schema/index.ts`.

## RLS Pattern
Tenant-scoped tables use PostgreSQL Row-Level Security:
```sql
ALTER TABLE {table} ENABLE ROW LEVEL SECURITY;
ALTER TABLE {table} FORCE ROW LEVEL SECURITY;
CREATE POLICY {table}_tenant_select ON {table}
  FOR SELECT USING (tenant_id = current_setting('app.tenant_id', true)::uuid);
```
RLS SQL is in `drizzle/0001_rls_policies.sql` (manual, not generated by drizzle-kit).

## SET LOCAL Pattern
Every tenant-scoped query must run inside a transaction with `SET LOCAL`:
```typescript
await db.transaction(async (tx) => {
  await tx.execute(sql`SET LOCAL app.tenant_id = ${tenantId}`);
  return tx.select().from(users);
});
```
The `true` parameter in `current_setting('app.tenant_id', true)` returns NULL instead of error when unset.

## Client Factory
```typescript
import { createDb } from "@bepro/db";
const db = createDb(databaseUrl);
```

## Commands
- `pnpm db:generate` — Generate migration SQL from schema changes
- `pnpm db:push` — Push schema directly to database (dev only)
- `pnpm db:studio` — Open Drizzle Studio for data inspection
- `pnpm db:exec <path>` — Apply a raw SQL file via Neon HTTP driver (supports `$$`-quoted functions)
- `pnpm db:query "SELECT …"` — Run an ad-hoc query and print as a table
- `pnpm db:create-app-worker` — One-time: create the `app_worker` role (NOBYPASSRLS) for integration tests + future production rotation

## Roles

| Role | Purpose | RLS |
|---|---|---|
| `neondb_owner` | Admin / migrations / seeds / fixture setup in tests | BYPASSRLS (Neon default) |
| `app_worker` | Worker runtime + integration tests that prove RLS | NOBYPASSRLS — policies actually enforced |

Integration tests open two Neon HTTP clients: one as `neondb_owner` to seed fixtures, one as `app_worker` to exercise the endpoint under RLS. The `app_worker` connection string lives in `apps/api/.dev.vars` as `DATABASE_URL_WORKER` (generated by the create script, never committed).

## Tables With RLS (tenant-scoped, `FORCE ROW LEVEL SECURITY`)

- `users`, `audit_events` — from initial migrations (`0001_rls_policies.sql`)
- `clients`, `client_assignments`, `client_contacts`, `client_positions`, `client_documents` — from clients module (`0002_rls_clients.sql`)
- `candidates`, `candidate_attachments`, `candidate_duplicate_links`, `rejection_categories`, `decline_categories`, `privacy_notices`, `retention_reviews` — from candidates module (`0005_candidates_rls.sql`)

All use the same policy shape: `tenant_id = current_setting('app.tenant_id', true)::uuid` for SELECT/INSERT/UPDATE, `USING (false)` for DELETE (soft-delete only, LFPDPPP).

Append-only tables (`audit_events`, `candidate_duplicate_links`, `retention_reviews`) additionally deny UPDATE/DELETE at the GRANT level for `app_worker` — defense in depth.

## Tables Without RLS
- `tenants` — Queried before tenant context is established (login)
- `refresh_tokens` — Accessed by token_hash, ownership via user_id FK
- `password_reset_tokens` — Pre-authentication artifact; no tenant context at issuance. Ownership enforced by user_id FK; isolation by the public endpoints exposing no tenant signal (feature 009 / ADR-009).
