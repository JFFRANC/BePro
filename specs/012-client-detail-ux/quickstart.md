# Quickstart — Feature 012: Client Detail UX + Contact Cargo + Candidate Base-Form

**Audience**: developer landing on this branch, ready to implement or review.

---

## 1. Prerequisites

- Working monorepo on branch `012-client-detail-ux` (already checked out).
- Node + pnpm installed (versions per repo root `package.json`).
- Local Neon connection string in `apps/api/.dev.vars` (`DATABASE_URL` for admin, `DATABASE_URL_WORKER` for `app_worker` integration tests).
- A second `DATABASE_URL` (admin/`neondb_owner`) available in your shell for the migration script.

---

## 2. Apply the database migration

```bash
# from repo root
pnpm --filter @bepro/db db:exec packages/db/drizzle/0011_client_description_contact_position.sql
```

Verify:

```bash
pnpm --filter @bepro/db db:query "SELECT column_name, data_type, is_nullable FROM information_schema.columns WHERE table_name='clients' AND column_name='description';"
pnpm --filter @bepro/db db:query "SELECT column_name, data_type, character_maximum_length FROM information_schema.columns WHERE table_name='client_contacts' AND column_name='position';"
```

Expected:

| table_name | column_name | data_type | is_nullable | char_max |
|---|---|---|---|---|
| clients | description | text | YES | n/a |
| client_contacts | position | character varying | YES | 120 |

## 3. Run the pre-deploy migration script (dry-run, then commit)

```bash
# Dry-run: prints the per-tenant rename plan, makes no writes
pnpm tsx scripts/012-rename-legacy-formconfig-collisions.ts --dry-run

# When the plan looks correct, run for real
pnpm tsx scripts/012-rename-legacy-formconfig-collisions.ts
```

The script:
- Connects with `DATABASE_URL` (admin / `neondb_owner` — bypasses RLS).
- For each tenant, in one transaction: rewrites `clients.form_config.fields[].key` collisions to `legacy_<key>` and rewrites `candidates.additional_fields[<key>] → additional_fields[legacy_<key>]`.
- Writes one `audit_events` row per tenant summarizing the renames.
- Idempotent — safe to re-run.

## 7. Manual verification flow (matches user stories)

### US1 — Description block

1. Log in as admin (or any tenant user).
2. Navigate to a client's detail page.
3. Click "Editar cliente" → fill `Descripción` with multi-line text including a markdown sample like `**bold** text\nLine 2`.
4. Save and reload.
5. Expect: the page shows the description with `\n` rendered as a visible line break and `**bold**` shown literally (no HTML interpretation).

### US2 — 2-column layout + clipboard

1. On the same page (≥ 1024px viewport), expect: map on the left at ~h-64, address line below the map, "Copiar ubicación" button below the address. Right column shows the description block then the general info card.
2. Resize the window below 768px (or open Chrome DevTools mobile emulator at 375px). Expect: single column, map at the bottom of the page.
3. Click "Copiar ubicación". Expect: a sonner success toast and the address pasted into a fresh browser tab's URL bar (or any text input).
4. Test the fallback: open the page over `http://` (non-secure) — clicking "Copiar ubicación" should show a sonner message including the address.

### US3 — Tab rename + redirect

1. As admin, view a client. Last tab reads "Formulario" (not "Configuración").
2. Visit `http://localhost:5173/clients/<id>/config` directly. Expect: URL replaces to `/clients/<id>` and the Formulario tab is selected (no 404).

### US4 — Base-form lock

1. Open the Formulario tab. Expect: 9 locked rows at the top with a "Campo base" badge; no delete/rename controls on those rows; admin custom fields appear below.
2. Try to add a custom field with key `fullName`. Expect: the form rejects with the Spanish collision message before submitting.
3. Open the candidate registration form. Expect: 9 base fields render in order. `recruiterName` is pre-filled with your full name (editable). `accountExecutiveName` is pre-filled with the client's primary AE (or empty if none).
4. Manually corrupt a tenant's `formConfig` from `db:studio` (don't do this against shared dev DBs!) by overwriting a base key. Submit a candidate-create. Expect: 500 with `form_config_tampered` and `missingBaseKeys` listing the gap.

### US5 — Contact position

1. Open the Contactos tab. Add a contact "Mariana López" with `position: "Recursos Humanos"`.
2. Expect: the contact row shows "Recursos Humanos" next to the name.
3. Edit the contact, change `position` to "Operaciones". Verify: an `audit_events` row exists with `diff.position = { old: "Recursos Humanos", new: "Operaciones" }`.

### US6 — Mobile

1. Open the page at 375px (Chrome DevTools iPhone SE). Expect: header → description → general info card → tabs → map (last). No horizontal scroll.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| API tests pass locally but fail in CI: 500 on candidate-create | Pre-deploy script not run on the CI seed DB | Add the script invocation to the CI seed step or seed `clients.form_config.fields[]` clean |
| `BASE_CANDIDATE_FIELDS is undefined` in web tests | Forgot to rebuild `@bepro/shared` | `pnpm --filter @bepro/shared build` |
| RLS test sees other tenant's description | `SET LOCAL` not active, or test using `neondb_owner` instead of `app_worker` | Verify `DATABASE_URL_WORKER` env var; constitution §I |
| "Copiar ubicación" toast appears but clipboard is empty in headless Playwright | Permission flag on browser context | Pass `permissions: ['clipboard-read','clipboard-write']` in the Playwright context |
| Layout tests flaky at 768px | CSS animations / transitions | Set `prefers-reduced-motion` in test setup |
